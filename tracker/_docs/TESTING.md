# Initial test concepts

Generated by Haiku.

---

**Unit Tests** (test components in isolation):

**JSON/Serialization layer:**

- Deserialize valid tracker.json → correct Issue objects
- Deserialize malformed JSON → error, don't crash
- Serialize Issue with special characters in title/notes → valid JSON roundtrip
- Serialize Issue with empty timeline → still valid
- Timestamp parsing (bad format, missing timezone, leap second edge cases)

**Status derivation (timeline → current status):**

- Empty timeline → error or default state?
- Single `opened` event → status is `open`
- Timeline ends with `blocked` → status is `blocked`
- Timeline has events out of chronological order → what do you do? (probably reject on load)
- Timeline with duplicate timestamps → allowed or error?

**Blocking logic:**

- Issue blocks itself (`blockedBy: ["1"]` on issue 1) → error
- Blocking cycle (1 blocks 2 blocks 3 blocks 1) → detected on load? on creation?
- Block a nonexistent issue → error
- Unblock when not currently blocked → allowed or error?

**ID generation:**

- Auto-increment ID collisions when loading + creating new
- ID types (string vs int) consistency across operations

**Filtering:**

- Filter with empty result set → return empty list, not error
- Multiple label filters (AND vs OR semantics—what did you decide?)
- Filter by status that doesn't exist (e.g., `--status foo`) → error
- Priority sort when priorities are missing → default sort

**Integration Tests** (real workflows):

**Full lifecycle:**

- Create issue → list shows it → mark in-progress → block it → unblock it → mark done → list still shows it (closed)
- Create, close, load from disk in new process → data persists

**Messy workflows:**

- Create 10 issues, delete 5 by ID, create 3 more → IDs are unique
- Create issue A with priority high, B with priority low, C with priority medium → list is high→medium→low
- Create issues with overlapping labels, filter by multiple labels → correct intersection
- Load JSON with missing issue in blockedBy array → error or skip?

**File I/O failure modes:**

- File doesn't exist yet → create it on first write
- File exists but is empty → error or default tracker?
- File is corrupted mid-write (process crashes) → can you recover or is it toast?
- Permission denied on read → helpful error, not panic
- Directory instead of file → error message

**Edge case workflows:**

- Create issue with zero-length title → reject
- Create issue with title that's only whitespace → reject
- Block an issue that's already in progress (state machine rules)
- Transition from done back to open → allowed or not? (design decision)
- Block with empty reason string → allowed or reject?
- Issue with 100+ timeline events → still serializes cleanly?

**The adversarial question**: What's your blocking model's strictness?

- Can you transition `done` → `open`? (probably not, but enforce it?)
- Can you block an already-blocked issue again? (probably allows duplicate block events?)
- Can you have duplicate labels on one issue? (probably deduplicate?)

These design decisions affect your test assertions.
